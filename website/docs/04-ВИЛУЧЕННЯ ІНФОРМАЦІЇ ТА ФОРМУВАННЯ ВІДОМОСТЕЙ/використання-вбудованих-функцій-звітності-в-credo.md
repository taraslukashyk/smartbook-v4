## Використання вбудованих функцій звітності в Credo Дороги та Топоматик Robur

ст. 81


Програмні комплекси Credo та Топоматик Robur традиційно мають сильний акцент на автоматизованому формуванні відомостей та специфікацій відповідно до стандартів, прийнятих в Україні та країнах СНД.

### A. Credo Дороги (та інші модулі CREDO):

*   **Система формування відомостей:** Credo має розгалужену систему генерації різноманітних відомостей, які є невід'ємною частиною проєктної документації.
    *   **Відомості по плану траси:** Координати вершин кутів, елементи кривих, розбивка пікетажу.
    *   **Відомості по поздовжньому профілю:** Відмітки, ухили, елементи вертикальних кривих, робочі відмітки.
    *   **Відомості по поперечних профілях:** Площі насипу/виїмки по кожному поперечнику, обсяги шарів дорожнього одягу.
    *   **Зведені відомості обсягів земляних мас:** Розрахунок за методом поперечників або за цифровими моделями.
    *   **Відомості елементів дорожнього одягу:** Обсяги матеріалів по шарах.
    *   **Відомості по елементах облаштування:** Кількість знаків, довжини огороджень, розмітки (якщо ці елементи були закладені в проєкт).
    *   **Відомості по штучних спорудах** (якщо використовується відповідний модуль, наприклад, Credo Труби).
*   **Налаштування шаблонів відомостей:** Користувач може налаштовувати шаблони відомостей (зовнішній вигляд, склад стовпців, шрифти) відповідно до вимог ДСТУ або корпоративних стандартів.
*   **Експорт відомостей:** Відомості можуть бути роздруковані безпосередньо з програми, збережені у внутрішніх форматах Credo, або експортовані в поширені формати:
    *   **RTF, DOC, DOCX (для Microsoft Word).**
    *   **XLS, XLSX (для Microsoft Excel).**
    *   **TXT, CSV (текстові формати).**
    *   **DWG, DXF (як графічні об'єкти або таблиці AutoCAD).**
*   **Інтеграція з кошторисними програмами:** Деякі версії Credo можуть мати можливість експорту даних у форматах, сумісних з популярними кошторисними програмами.

### B. Топоматик Robur – Автомобільні Дороги (та інші модулі):

*   **Аналогічний підхід до Credo:** Robur також орієнтований на комплексне проєктування та автоматизований випуск документації, включаючи відомості обсягів робіт.
*   **Типи відомостей:**
    *   Повний набір відомостей, необхідних для проєктування автомобільних доріг: по трасі, профілю, поперечниках, земляних масах, дорожньому одягу, елементах облаштування.
    *   Спеціалізовані відомості для проєктування залізниць (у відповідних модулях).
*   **Розрахунок обсягів:** Вбудовані алгоритми для точного розрахунку обсягів земляних робіт (з урахуванням коефіцієнтів розпушення/ущільнення), матеріалів дорожнього одягу, розкривних робіт.
*   **Налаштування та експорт:**
    *   Можливість налаштування зовнішнього вигляду та змісту відомостей.

**Загальні особливості для Credo та Robur:**

*   **Орієнтація на стандарти:** Ці програми розроблялися з урахуванням вимог ГОСТ та ДСТУ до оформлення проєктної документації, що спрощує процес підготовки відомостей.
*   **Інтегрований підхід:** Розрахунок обсягів та формування відомостей тісно пов'язані з процесом проєктування. Зміни в геометрії траси або конструкції автоматично відображаються у відомостях (після перерахунку).
*   **Обмеження гнучкості:** Порівняно з Civil 3D + Dynamo, можливості для створення повністю користувацьких звітів або вилучення дуже специфічних даних можуть бути більш обмеженими. Робота з атрибутивною інформацією, не пов'язаною безпосередньо з розрахунком стандартних обсягів, може бути менш розвиненою.

### 4.2.3. Налаштування форматів експорту даних (наприклад, в CSV, XML, таблиці DWG)

Після того, як дані про обсяги та кількості розраховані в основному моделюючому програмному забезпеченні, наступним важливим кроком є їх експорт у формати, зручні для подальшої обробки, аналізу, використання в інших програмах (наприклад, Excel, кошторисні системи) або для включення в текстову частину проєктної документації.

**Загальні принципи налаштування експорту:**

*   **Вибір відповідного формату:** Формат експорту обирається залежно від подальших цілей використання даних.
    *   **CSV (Comma-Separated Values) / TXT (Text Files):** Прості текстові формати, де дані розділені комами, крапками з комою або табуляцією. Легко імпортуються в Excel та інші табличні процесори, бази даних. Добре підходять для передачі "сирих" табличних даних.
    *   **XLS / XLSX (Microsoft Excel):** Дозволяють експортувати дані безпосередньо у формат електронних таблиць Excel, зберігаючи при цьому деяке форматування (якщо підтримується ПЗ). Найбільш зручний формат для подальших розрахунків та обробки.
    *   **XML (Extensible Markup Language):** Структурований текстовий формат, який часто використовується для обміну даними між різними програмними системами. Дозволяє передавати не тільки самі дані, але й інформацію про їхню структуру. Може використовуватися для інтеграції з кошторисними програмами або базами даних.
    *   **HTML (HyperText Markup Language):** Формат веб-сторінок. Деякі ПЗ дозволяють експортувати звіти в HTML для зручного перегляду в браузері.
    *   **DOC / DOCX (Microsoft Word) / RTF (Rich Text Format):** Використовуються для експорту вже сформованих відомостей та звітів, які можна потім редагувати в текстовому редакторі.
    *   **DWG / DXF (AutoCAD Drawing / Drawing Exchange Format):** Дані можуть бути експортовані як таблиці AutoCAD, розміщені безпосередньо на кресленнях, або як

графічні об'єкти.

* **Вибір даних для експорту:** Не завжди потрібно експортувати всі доступні дані. Слід обирати лише ті стовпці та рядки, які необхідні для конкретного завдання, щоб уникнути надлишковості.
* **Налаштування роздільників та кодування (для CSV/TXT):** Важливо правильно вказати символ-роздільник (кома, крапка з комою, табуляція) та кодування текстового файлу (наприклад, UTF-8, Windows-1251), щоб забезпечити коректне зчитування даних в інших програмах, особливо при роботі з кириличними символами.
* **Збереження структури та форматування:** По можливості, слід намагатися зберегти структуру таблиць (заголовки, стовпці) та базове форматування при експорті в XLS/XLSX або DOC/DOCX.
* **Використання шаблонів експорту:** Деякі програми дозволяють створювати та зберігати шаблони налаштувань експорту для часто використовуваних типів звітів, що прискорює роботу.

### Налаштування експорту в Autodesk Civil 3D:

* **Звіти з Reports Manager:**
    * При генерації звіту (наприклад, звіту по трубах або трасі) зазвичай надається можливість вибору формату експорту (HTML, XML, CSV, DOC, XLS тощо) безпосередньо в діалоговому вікні менеджера звітів.
    * Для деяких форматів (наприклад, XML) можуть використовуватися XSL-файли (таблиці стилів), які визначають структуру та вигляд експортованого файлу. Модифікація XSL-файлів дозволяє гнучко налаштовувати вихідний формат, але вимагає спеціальних знань.
* **Таблиці Civil 3D:**
    * Дані з таблиць, розміщених у кресленні, можна експортувати через стандартні функції AutoCAD: команда **TABLEEXPORT** дозволяє зберегти таблицю у форматі CSV.
    * Також можна просто скопіювати вміст таблиці з креслення та вставити його в Excel.
* **Quantity Takeoff Manager:**
    * Звіти QTO можуть бути експортовані у формати HTML, CSV, XML. Налаштування експорту зазвичай обмежені вибором формату.
* **Volumes Dashboard:**
    * Таблицю обсягів можна вставити в креслення, а потім експортувати як таблицю AutoCAD, або скопіювати дані. Також є опція "Generate Cut/Fill Report", яка створює HTML-звіт.
* **Dynamo для Civil 3D:**
    * Надає найширші можливості для налаштування експорту. За допомогою вузлів (nodes) для роботи з файлами (наприклад, "Export Excel", "Export CSV") можна точно визначити, які дані, в якому порядку та з яким форматуванням будуть експортовані. Можна створювати багатосторінкові книги Excel, форматувати комірки, додавати формули тощо.

### 4.2.4. Експорт первинних даних про об'єми та кількості

Після налаштування форматів виконується безпосередній експорт даних. Важливо розуміти, що саме експортується — це "первинні" дані, отримані безпосередньо з моделі або розрахунків всередині ПЗ.

**Типовий склад експортованих первинних даних:**

*   **Для земляних робіт:**
    *   Обсяги насипу та виїмки попікетно або по ділянках.
    *   Площі зняття рослинного шару.
    *   Обсяги розкривних робіт.
    *   Зведені дані про баланс земляних мас.
*   **Для дорожнього одягу:**
    *   Площі та обсяги кожного конструктивного шару (покриття, основа, додаткові шари) попікетно або для всієї ділянки.
    *   Типи матеріалів шарів.
*   **Для лінійних елементів:**
    *   Довжини бордюрів (за типами).
    *   Довжини лотків, канав (за типами та розмірами).
    *   Довжини трубопроводів (за діаметрами та матеріалами).
    *   Довжини огороджень (за типами).
*   **Для штучних елементів:**
    *   Кількість колодязів (за типами).
    *   Кількість дорожніх знаків (за типами та типорозмірами).
    *   Кількість опор освітлення.
*   **Для атрибутивних даних (якщо експортуються окремо, наприклад, через Property Sets або Dynamo):**
    *   Таблиці, де кожен рядок відповідає елементу моделі, а стовпці — його

атрибутам (ID, найменування, матеріал, розміри, код за класифікатором тощо).

**Процес експорту:**

1. **Вибір об'єктів або звіту:** У програмі обираються об'єкти, для яких потрібно вилучити дані, або запускається генерація відповідного звіту/відомості.
2. **Запуск команди експорту:** Використовується відповідна функція експорту ("Зберегти як", "Експорт", "Generate Report").
3. **Вибір формату та налаштувань:** Вказується бажаний формат файлу та параметри експорту (як описано в п. 4.2.3).
4. **Вказання імені та місця збереження файлу:** Файл зберігається у відповідну папку проєктної структури (наприклад, у підпапку "Експорт_з_Моделі" всередині папки "Відомості" або "Розрахунки"). Дотримуйтесь корпоративних правил іменування файлів.
5. **Перевірка експортованого файлу:** Після експорту обов'язково відкрийте файл у цільовій програмі (наприклад, Excel, Word) та перевірте:
    * Чи всі дані експортувалися коректно.
    * Чи немає спотворень у форматуванні або кодуванні.
    * Чи відповідають дані очікуваним значенням.

Збереження первинних експортованих даних є важливим, оскільки вони слугують вихідною точкою для подальших розрахунків та формування фінальних відомостей. Це дозволяє відстежити походження цифр та, за потреби, повернутися до вихідних даних з моделі.

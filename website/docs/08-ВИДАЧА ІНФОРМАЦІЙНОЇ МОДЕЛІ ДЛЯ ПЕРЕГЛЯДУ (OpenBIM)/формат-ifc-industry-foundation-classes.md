## Формат IFC (Industry Foundation Classes)

Формат IFC є наріжним каменем концепції OpenBIM та основним інструментом для забезпечення інтероперабельності при обміні даними інформаційних моделей. Розуміння його структури, можливостей та обмежень є важливим для коректної видачі та використання ІМ.

### 8.2.1. Опис формату IFC як відкритого стандарту

*   **Визначення:** IFC (Industry Foundation Classes) — це відкритий, об'єктно-орієнтований формат файлу, розроблений міжнародною організацією buildingSMART (раніше відома як International Alliance for Interoperability, IAI) для опису даних будівельної галузі та забезпечення обміну інформацією між різними програмними додатками.
*   **Стандарт ISO:** Специфікація IFC є міжнародним стандартом ISO 16739.
*   **Об'єктно-орієнтована структура:** Дані в IFC організовані у вигляді об'єктів (наприклад, стіна, балка, труба, дорога), кожен з яких має свою геометрію, атрибути (властивості) та взаємозв'язки з іншими об'єктами.
*   **Схема даних:** IFC базується на чітко визначеній схемі даних (IFC Schema), яка описує класи об'єктів, їхні атрибути та типи взаємозв'язків. Схема розроблена на основі мови EXPRESS.
*   **Формати файлів:** IFC-дані можуть зберігатися у декількох форматах файлів:
    *   **.ifc:** Текстовий формат, що базується на STEP (Standard for the Exchange of Product model data - ISO 10303-21). Файли .ifc можна відкрити та прочитати в текстовому редакторі (хоча їх структура досить складна для ручного аналізу).
    *   **.ifcXML:** Формат, що базується на XML (Extensible Markup Language).
    *   **.ifcZIP:** ZIP-архів, що містить файл .ifc або .ifcXML, що дозволяє зменшити розмір файлу.
*   **Нейтральність до постачальника ПЗ:** Формат IFC не належить жодному конкретному розробнику програмного забезпечення, що забезпечує його незалежність та широку підтримку в галузі.
*   **Призначення:** IFC призначений для опису інформації про будівлі та інфраструктурні об'єкти протягом усього їхнього життєвого циклу — від концепції та проєктування до будівництва, експлуатації та демонтажу.

### 8.2.2. Переваги використання IFC для обміну даними

Використання формату IFC для обміну інформаційними моделями надає низку суттєвих переваг:

1. **Інтероперабельність:** Дозволяє різним програмним продуктам від різних виробників обмінюватися даними BIM-моделей, зменшуючи проблеми сумісності.
2. **Збереження даних:** Забезпечує передачу не тільки 3D-геометрії, але й великого обсягу атрибутивної інформації (властивостей), пов'язаної з елементами моделі.
3. **Спільна робота (Collaboration):** Спрощує співпрацю між різними учасниками проєкту (архітекторами, конструкторами, інженерами Мереж, кошторисниками, будівельниками, експлуатуючими організаціями), які можуть використовувати різне ПЗ.
4. **Контроль якості та координація:** IFC-моделі можуть бути використані в спеціалізованих програмах для перевірки на колізії, відповідність стандартам та вимогам (наприклад, Solibri Model Checker, Autodesk Navisworks).
5. **Довгострокове архівування:** Як відкритий стандарт, IFC краще підходить для довгострокового зберігання проєктних даних, оскільки його специфікація є загальнодоступною і не залежить від майбутнього конкретних комерційних форматів.
6. **Незалежність від програмного забезпечення:** Замовники та інші зацікавлені сторони можуть отримувати доступ до інформації про проєкт, не маючи потреби купувати те саме дороге ПЗ, в якому модель була створена. Існує багато безкоштовних програм для перегляду IFC-файлів.
7. **Підтримка життєвого циклу об'єкта:** IFC дозволяє передавати інформацію, необхідну для різних етапів – від проєктування до експлуатації та управління активами.
8. **Прозорість процесів:** Використання відкритого формату підвищує прозорість обміну даними та прийняття рішень.

### 8.2.3. Версії IFC та їх особливості

Специфікація IFC постійно розвивається для кращого задоволення потреб будівельної галузі. Існує декілька основних версій IFC, кожна з яких має свої особливості:

* **IFC1.x, IFC2.0, IFC2x, IFC2x2:** Ранні версії, які зараз практично не використовуються.
* **IFC2x3 (випущена у 2006 році):**
    * Довгий час була найпоширенішою та найбільш стабільно підтримуваною версією IFC у більшості програмних продуктів.
    * Має декілька "Model View Definitions" (MVD), таких як Coordination View, Structural Analysis View.
    * Добре підходить для координації моделей будівель, передачі базової геометрії та атрибутів.
    * **Coordination View 2.0 (CV 2.0):** Найбільш часто використовуваний MVD для IFC2x3, призначений для координації між різними дисциплінами.
* **IFC4 (випущена у 2013 році, також відома як ISO 16739:2013):**
    * Значно розширена та покращена версія порівняно з IFC2x3.
    * **Основні нововведення та покращення:**
        * **Нові та покращені класи об'єктів:** Додано багато нових класів для кращого опису різних типів будівельних та інфраструктурних елементів.
        * **Покращена підтримка геометрії:** Більш досконалі способи представлення

складної геометрії (наприклад, NURBS-поверхні, теселяція).
*   **Розширена підтримка інфраструктурних об'єктів (Infrastructure Extensions):** Хоча повноцінна підтримка інфраструктури (дороги, мости, тунелі) активно розвивається в наступних версіях (IFC 4.1, 4.2, 4.3), IFC4 заклав основу для цього.
*   **Нові MVD:**
    *   **IFC4 Reference View (RV):** Призначений для передачі геометрії та властивостей для координації, перегляду, аналізу колізій. Зазвичай забезпечує більш "чисту" геометрію. **Рекомендується для видачі моделі для перегляду та візуалізації.**
    *   **IFC4 Design Transfer View (DTV):** Призначений для передачі моделі з більшою кількістю параметричної інформації та взаємозв'язків між елементами, для можливості подальшого редагування в іншому ПЗ (хоча "round-trip" редагування через IFC все ще є складним завданням).
*   **Покращена робота з властивостями та класифікаторами.**

*   **IFC4.1 (IFC4 Addendum 1, випущена у 2018):**
    *   Включає розширення для інфраструктурних об'єктів, зокрема для мостів (IfcBridge).
*   **IFC4.2 (IFC4 Addendum 2, випущена у 2019):**
    *   Подальший розвиток підтримки мостів та геоТехніки.
*   **IFC4.3 (випущена у 2022 році, також відома як ISO 16739:2022):**
    *   **Значне розширення для інфраструктури:** Включає повноцінну підтримку для доріг (IfcRoad), залізниць (IfcRailway), портів та водних шляхів, тунелів (IfcTunnel). Це дуже важлива версія для інфраструктурних проєктів.
    *   Стандартизація опису трас (alignments), поперечних перерізів, конструктивних шарів.
    *   На момент написання цієї методички (травень 2025 року), підтримка IFC4.3 у комерційних програмних продуктах активно впроваджується, але може бути ще не повною у всіх системах.
*   **Майбутні версії (IFC5 і далі):** Розвиток IFC триває, з фокусом на подальшому розширенні підтримки різних типів об'єктів, покращенні семантичної інформації та інтеграції з іншими стандартами.

### Рекомендації щодо вибору версії IFC:

*   **Для сумісності з максимальною кількістю ПЗ (особливо старіших версій):** IFC2x3 Coordination View 2.0 все ще може бути робочим варіантом, але він має обмеження для інфраструктурних об'єктів.
*   **Для сучасних проєктів та кращої передачі геометрії та даних:** Рекомендується використовувати **IFC4 Reference View (RV)**. Ця версія добре підтримується багатьма сучасними програмами для перегляду та аналізу.
*   **Для інфраструктурних проєктів (дороги, мости, тунелі):** Якщо програмне забезпечення (як експортер, так і імпортер/переглядач) підтримує **IFC4.3**, це є найкращим вибором, оскільки ця версія спеціально розроблена для опису інфраструктурних об'єктів.
*   **Завжди уточнюйте вимоги замовника або приймаючої сторони:** Яку версію IFC та який MVD вони очікують отримати.
*   **Тестуйте експорт та імпорт:** Перед фінальною видачею моделі завжди тестуйте

експортований IFC-файл, відкриваючи його в декількох програмах для перегляду, щоб переконатися в коректності передачі геометрії та даних.

## Насичення IFC-моделі Інформацією

Як неодноразово підкреслювалося, справжня цінність інформаційної моделі полягає не лише в її тривимірній геометрії, а й у тій негеометричній (атрибутивній) інформації, яка прив'язана до кожного її елемента. При експорті моделі у формат IFC важливо забезпечити, щоб вся необхідна інформація була коректно передана. Рівень інформаційного насичення (LOI – Level of Information) повинен відповідати цілям видачі моделі та вимогам Замовника (EIR) або наступних етапів життєвого циклу об'єкта.

### 8.5.1. Яка інформація має бути обов'язково включена до IFC-моделі

Перелік обов'язкової інформації може варіюватися залежно від типу інфраструктурного об'єкта, стадії проєктування та конкретних завдань, для яких видається IFC-модель (наприклад, для координації, для передачі на будівництво, для експлуатації). Однак, існує базовий набір даних, який, як правило, є необхідним.

**А. Загальні ідентифікаційні дані для всіх елементів:**

*   **GlobalId (Глобальний унікальний ідентифікатор):** Автоматично генерується при експорті, забезпечує унікальність кожного екземпляра об'єкта в межах IFC-файлу та між різними файлами.
*   **Name (Ім'я/Найменування):** Зрозуміле ім'я елемента (наприклад, "Опора мосту №3", "Ділянка покриття ПК0-ПК1", "Труба зливової каналізації D300"). Може відповідати маркуванню на кресленнях.
*   **ObjectType (Тип об'єкта) / PredefinedType (Передвстановлений тип):** Уточнює тип

*   **IfcClass (Клас IFC):** Тип об'єкта за схемою IFC (наприклад, IfcRoad, IfcBridgePart, IfcPipeSegment, IfcManhole, IfcBuildingElementProxy для об'єктів, що не мають точного відповідника).
*   **Tag (Позначка/Марка):** Унікальне маркування елемента в межах проєкту, якщо таке використовується.
*   **Description (Опис):** Додатковий текстовий опис елемента, якщо необхідно.

### **B. Геометричні характеристики (багато з них розраховуються та експортуються автоматично):**

*   **Об'єми (Volumes):** Наприклад, об'єм бетону для конструкцій, об'єм шарів дорожнього одягу, об'єм земляних мас (якщо земляне полотно представлене як 3D-тіла).
*   **Площі (Areas):** Наприклад, площа поверхні шару покриття, площа укосу, площа фарбування.
*   **Довжини (Lengths):** Наприклад, довжина ділянки дороги, довжина труби, довжина бордюру.
*   **Розміри (Dimensions):** Основні габаритні розміри елементів (ширина, висота, товщина, діаметр).

### **C. Інформація про матеріали:**

*   **Material Name (Назва матеріалу):** Детальна назва матеріалу (наприклад, "Асфальтобетон щільний дрібнозернистий типу Б марки II", "Бетон важкий класу С20/25 (В25)").
*   **Material Layer (Шар матеріалу) (для багатошарових конструкцій):** Якщо елемент складається з декількох шарів (наприклад, дорожній одяг, стіна), інформація про кожен шар (назва матеріалу, товщина).
*   **Посилання на стандарт/норматив на матеріал:** (наприклад, ДСТУ, ГОСТ, ТУ).
*   **Фізико-механічні властивості (за потреби):** Щільність, міцність, морозостійкість (якщо це важливо для подальшого використання моделі).

### **D. Конструктивні та технічні характеристики:**

*   **Тип конструкції/елемента:** (наприклад, "Балка прогонової будови", "Стійка опори", "Колодязь оглядовий ККС-1").
*   **Несуча здатність, клас навантаження:** (наприклад, для люків, решіток, мостових конструкцій).
*   **Діаметр, ухил, пропускна здатність:** (для трубопроводів).
*   **Типорозмір, номер за ДСТУ:** (для дорожніх знаків).
*   **Висота, тип, матеріал:** (для опор освітлення, огороджень).

### Як визначається необхідний набір інформації?

*   **Вимоги Замовника (EIR - Employer's Information Requirements):** Замовник може чітко визначити, яка інформація повинна міститися в IFC-моделі на різних етапах.
*   **План виконання BIM-проєкту (BEP - BIM Execution Plan):** У BEP деталізуються вимоги до інформаційного насичення моделі, узгоджені між усіма учасниками проєкту.
*   **Призначення моделі:** Набір даних буде різним для моделі, що передається для координації, для кошторисних розрахунків, для будівництва або для експлуатації.
*   **Нормативні вимоги:** Деякі нормативні документи можуть встановлювати мінімальні вимоги до складу інформації в електронній моделі.

### Важливо забезпечити, щоб інформація була:

*   **Актуальною:** Відповідала останній версії проєктних рішень.
*   **Достовірною:** Не містила помилок або неточностей.
*   **Повною:** Містила всі необхідні дані для цільового використання.
*   **Структурованою:** Організована у вигляді логічних наборів властивостей (Property Sets) зі зрозумілими іменами атрибутів.

### 8.5.2. Як перевірити наявність та коректність атрибутів в IFC-переглядачі

Після експорту моделі в IFC та її відкриття в програмі для перегляду (наприклад, BIMvision, Solibri Anywhere) необхідно ретельно перевірити, чи вся запланована атрибутивна інформація була передана коректно.

1.  Оберіть декілька типових елементів кожної категорії (наприклад, ділянку асфальтобетонного покриття, бордюр, трубу, колодязь, опору мосту, дорожній знак).
    *   Обирайте як прості, так і складні елементи.

2.  **Перегляд властивостей (Property Sets):**
    *   У програмі-переглядачі активуйте вікно властивостей (зазвичай воно відкривається автоматично при виборі об'єкта або через спеціальну команду).
    *   **Перевірте наявність усіх очікуваних наборів властивостей (Pset_...).** Наприклад, для стіни можуть бути Pset_WallCommon, Pset_QuantityTakeOff, а також користувацькі набори властивостей, створені в основному ПЗ.
    *   **Розгорніть кожен набір властивостей та перевірте наявність усіх запланованих атрибутів** (порівнюючи зі списком атрибутів, визначеним у BEP або вимогах).

3.  **Перевірка значень атрибутів:**
    *   **Коректність значень:** Чи відповідають значення атрибутів тим, що були задані в основній моделі? Чи немає помилок, одруків, неправильних одиниць виміру?
    *   **Повнота заповнення:** Чи заповнені всі обов'язкові атрибути? Чи немає порожніх значень там, де вони повинні бути?
    *   **Відповідність типу даних:** Чи відповідає тип даних значенню (наприклад, для атрибута "Довжина" повинно бути числове значення, для "Матеріал" – текстове).
    *   **Логічна узгодженість:** Чи узгоджуються значення різних атрибутів між собою (наприклад, чи відповідає назва матеріалу його детальним характеристикам)?

4.  **Перевірка одиниць виміру:**
    *   Зверніть увагу на одиниці виміру для числових атрибутів (довжини, площі, об'єми). Чи відповідають вони тим, що використовуються в проєкті (зазвичай метри, $м^2$, $м^3$)? Деякі IFC-переглядачі можуть відображати одиниці, задані в IFC-файлі.

5.  **Перевірка ідентифікаторів:**
    *   Чи присутній атрибут GlobalId і чи є він унікальним?
    *   Чи коректно передані Name, ObjectType, Tag?

6.  **Перевірка класифікаційних кодів (якщо використовуються):**
    *   Чи присутні коди класифікаторів і чи відповідають вони очікуваним значенням?

7.  **Тестування пошуку та фільтрації:**
    *   Спробуйте знайти елементи за значенням певного атрибута або відфільтрувати модель за певним критерієм (наприклад, показати всі труби з певним діаметром або всі елементи з певним матеріалом). Це допоможе перевірити, чи коректно структуровані дані для аналізу.

8.  **Використання спеціалізованих інструментів валідації (за потреби):**
    *   Програми типу Solibri Model Checker дозволяють створювати правила для автоматизованої перевірки наявності та коректності атрибутивних даних (наприклад, "всі елементи типу IfcPipeSegment повинні мати заповнений атрибут Діаметр зі значенням більше 0").

1. Повернутися до основного моделюючого ПЗ.
2. Перевірити, чи правильно задані атрибути (Property Sets) для відповідних об'єктів.
